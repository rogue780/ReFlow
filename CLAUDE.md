# ReFlow Compiler — CLAUDE.md

This file defines the context, rules, and conventions for working on the ReFlow compiler. Read it fully at the start of every session. The rules here are not suggestions.

---

## What This Project Is

ReFlow is a strongly-typed functional language with composition-first design, linear ownership, and first-class streaming. This repository contains its reference compiler, written in Python, which emits C and links via clang. The Python compiler is temporary scaffolding — it exists to compile a self-hosted ReFlow compiler (Epic 11), after which it is archived.

---

## The Pipeline

Information flows forward through exactly these stages, in this order:

```
ReFlow source (.reflow)
  → compiler/lexer.py        → list[Token]
  → compiler/parser.py       → Module  (AST)
  → compiler/resolver.py     → ResolvedModule
  → compiler/typechecker.py  → TypedModule
  → compiler/lowering.py     → LModule
  → compiler/emitter.py      → str  (C source)
  → compiler/driver.py       → shells out to clang → binary
```

No pass reaches backward. No pass calls a later pass. If a design requires this, the design is wrong.

---

## File Responsibilities

Each file owns exactly one concern. Do not add logic that belongs to another file.

| File | Owns |
|------|------|
| `compiler/lexer.py` | source string → `list[Token]`. No semantics. |
| `compiler/parser.py` | `list[Token]` → `Module` AST. No type information. |
| `compiler/ast_nodes.py` | Dataclass definitions only. No logic, no imports from other compiler modules. |
| `compiler/resolver.py` | Binds names to `Symbol` objects. No type inference. |
| `compiler/typechecker.py` | Infers and verifies types. No C-level concerns. |
| `compiler/lowering.py` | Typed AST → `LModule`. No C syntax. No formatting. |
| `compiler/emitter.py` | `LModule` → C string. No decisions. Pure formatting. |
| `compiler/mangler.py` | ReFlow names → C identifiers. Nothing else. |
| `compiler/errors.py` | Error type definitions only. No logic. |
| `compiler/driver.py` | Pipeline orchestration. No compiler logic. |
| `runtime/reflow_runtime.h` | Runtime type and function declarations. |
| `runtime/reflow_runtime.c` | Runtime implementations. |

## Name Mangling — DO NOT DEVIATE

All C identifiers generated by the compiler come from `compiler/mangler.py`. The rules are:

```
Prefix:   rf_
Dots:     math.vector     → rf_math_vector
Type:     math.vector.Vec3    → rf_math_vector_Vec3
Method:   math.vector.Vec3.dot  → rf_math_vector_Vec3_dot
Function: pipeline.orders.run   → rf_pipeline_orders_run
```

**Never** construct a mangled name with string formatting outside `mangler.py`. If you need a new generated name form (closure frames, stream state machines, etc.), add a function to `mangler.py`.

The only exception: temp variables within a single function (`_rf_tmp_1`, etc.) are produced by the emitter's `fresh_temp()` method and are locally scoped.

---

## AST Nodes Are Immutable After Parsing

`ast_nodes.py` nodes are frozen once the parser produces them. Downstream passes annotate via side maps:

- `resolver.py` builds `ResolvedModule.symbols: dict[ASTNode, Symbol]`
- `typechecker.py` builds `TypedModule.types: dict[ASTNode, Type]`
- `lowering.py` reads both and produces a new `LModule`

Never add fields to AST node dataclasses after Epic 3. Never mutate a field on an existing node.

---

## Error Handling

All compiler errors must be raised as subclasses of `ReFlowError` from `compiler/errors.py`:

- `LexError` — bad character, unterminated literal
- `ParseError` — syntax error
- `ResolveError` — undefined name, scope violation
- `TypeError` — type mismatch, exhaustiveness, purity
- `EmitError` — malformed LIR

Every error requires `message`, `file`, `line`, `col`. Never use `raise ValueError`, `raise Exception`, `print` + `exit`, or bare `assert` in compiler code.

## The Spec Is the Source of Truth

The ReFlow language specification is at `reflow_spec.md`. When implementing any language feature:

1. Find the relevant spec section.
2. Read it fully, including cross-references.
3. Only then write code.

If the spec does not address a behavior, do not invent one. Use the safest option (reject with a clear error) and add a `# SPEC GAP: <description>` comment.

## Project Plan

The full build plan is at `reflow_compiler_plan.md` in the project root. It contains
all epics, stories, and tickets with their IDs (e.g., RT-6-4-1) and blocker flags.

Before starting work on any ticket:
1. Read the relevant story in the plan to understand the full scope
2. Check that all [BLOCKER] tickets in previous stories are complete
3. Use the ticket ID in your commit message

The dependency map at the bottom of the plan defines which epics must be
complete before others can begin. Do not start an epic whose dependencies
are unfinished.

**The five most common places where "obvious" diverges from the spec:**

1. `:imut` accepts any binding; `:mut` requires a `:mut` binding (asymmetric, not symmetric)
2. Option auto-lifting fires only when the target type is statically known and the source is not generic
3. Struct spread (`..source`) bypasses constructors — never route it through a constructor
4. `ex.data` is mutable in retry blocks; `ex.original` is always immutable and set once at throw time
5. Stream single-consumer: consumed once means consumed forever in the current scope

## Testing — A Ticket Is Not Done Without This

Before marking any ticket complete:

1. `make test` must pass with zero failures.
2. At least one positive test covers the new behavior.
3. At least one negative test covers the error case if any exists.

Test structure:
- `tests/unit/` — unit tests for individual pipeline stages
- `tests/programs/` + `tests/expected/` — golden file tests for the emitter
- `tests/programs/` + `tests/expected_stdout/` — end-to-end execution tests
- `tests/programs/errors/` + `tests/expected_errors/` — negative compile tests

Golden files are updated deliberately and reviewed before committing. Never auto-regenerate without reading the diff.

## The Bootstrap Constraint

The Python compiler will eventually compile itself. This means:

- Every design decision in the compiler pipeline must have a ReFlow-language equivalent
- Do not use Python-specific tricks (introspection, arbitrary-precision integer math, MRO) inside the pipeline
- Do use Python `match` for AST node dispatch — it maps cleanly to ReFlow `match` on sum types
- The runtime (`reflow_runtime.h/c`) is permanent and exempt from this constraint

## Known Implementation Decisions

These decisions were made during Epics 7–9 and are non-obvious. The self-hosted
compiler (Epic 11) must replicate them exactly.

1. **Static member access** — `Type.static_field` lowers to the mangled global
   variable name (`rf_module_Type_field`), not a field access on the type name.
   The resolver binds the `FieldAccess` node with `SymbolKind.STATIC`; the
   lowering checks this and emits `LVar(mangled_name)`.

2. **Static string fields** — `RF_String*` globals cannot be initialized at
   compile time (because `rf_string_from_cstr` is a function call). They are
   declared as `NULL` and initialized in a generated `_rf_init_statics()`
   function called from `main()` before the ReFlow entry point.

3. **Result struct layout** — Result types use flat fields (`tag`, `ok_val`,
   `err_val`) at the top level of the struct. There is no `.payload` union
   wrapper. The runtime's `RF_RESULT_TYPE` macro uses a union, but the compiler
   generates its own struct definitions with flat fields instead.

4. **Option auto-lifting** — When a `let` binding has an `option<T>` type
   annotation but the value expression is a plain `T`, the lowering wraps the
   value in an `RF_SOME` compound literal (`{.tag = 1, .value = expr}`) at the
   let-binding site. The type checker detects the lift but does not insert AST
   nodes (AST is immutable); the lowering performs the actual wrapping.

5. **`none` literal type** — `NoneLit` uses the enclosing function's return
   type (via `_current_fn_return_type`) to determine the concrete option struct
   (e.g., `RF_Option_int`). Without this, it defaults to `RF_Option_ptr` which
   causes type mismatches for value-type options.

6. **`ok()`/`err()` result type** — The type checker assigns `TResult(T, TAny)`
   to `ok(value)` and `TResult(TAny, E)` to `err(value)`. The lowering uses
   `_current_fn_return_type` (the fully-resolved return type) instead, so the
   compound literal uses the correct concrete struct name.

## MCP Servers

### Sequential Thinking
Use this before implementing anything in Epics 5–8 (resolver, type checker,
lowering, emitter) where the implementation requires reasoning through a chain
of dependencies. Specifically, invoke it when:
- Implementing a type inference rule that affects downstream lowering
- Designing how a ReFlow construct maps through the full pipeline to C
- Debugging a test failure where the root cause could be in any of several passes
- Planning a change that touches more than two files

Do not invoke it for mechanical tasks: adding a token type to the lexer,
writing a dataclass definition, adding a golden file test.

### GitHub
Use for all branch and PR operations. The workflow is:
- One branch per story: `story/RT-<epic>-<story>-<short-name>`
- Open a PR when all tickets in a story are complete and tests pass
- Never commit directly to main

Also use it to check if a ticket is already in progress on another branch
before starting work.

### Filesystem
Use when you need to search across the codebase for usages before making
a change. Specifically: before modifying a function signature, before
renaming anything, before changing a data structure that other passes depend
on. Search for all call sites first.

### Shell
Use to run `make test` before marking any ticket complete. Also use to
run a specific test in isolation when debugging:
`python -m pytest tests/unit/test_<module>.py -v`

Never mark a ticket done without having run the tests through this server
and seen them pass.

## Commit Discipline

- One logical change per commit.
- Commit message references the ticket ID: `RT-6-4-1: exhaustiveness check for option<T>`.
- Never commit with failing tests.
- Emitter changes and golden file updates go in the same commit.
- Spec gap comments (`# SPEC GAP:`) are committed and not removed until the spec is updated.

## What To Do When Uncertain

1. Check `reflow_spec.md` first.
2. Check `CLAUDE.md` (this file) second.
3. Check the relevant skill file third.
4. If still uncertain, make the safest choice (reject with an error) and leave a `# TODO: confirm with spec` comment.

Do not guess at semantics. Do not implement what seems obvious when the spec defines something specific.

## Environment

This project uses a Python virtual environment and nvm for Node. The `.envrc`
file handles both automatically when you enter the project directory, but all
shell commands must respect these:

**Python:** Always use `python` or `pytest` directly — the venv is activated
by direnv, so the correct Python is already on PATH. Never use `python3` or
`/usr/bin/python3` explicitly, as these bypass the venv.

**Node/npx:** nvm is loaded by `.envrc`. If a shell command fails to find
`node` or `npx`, run `source ~/.nvm/nvm.sh` first. Never install global npm
packages with sudo.

**Running tests:** Always run tests via:
```bash
make test
```
or for a specific test:
```bash
pytest tests/unit/test_.py -v
```

Never run `python -m pytest` with an explicit python path. The venv Python
handles this.

**Verifying the environment is correct before any shell command:**
```bash
which python   # should point to .venv/bin/python
which node     # should point to ~/.nvm/versions/node/v22.x.x/bin/node
```

If either points somewhere unexpected, do not proceed — the environment
is not set up correctly and commands will fail or use wrong versions.
