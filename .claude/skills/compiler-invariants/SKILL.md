# Flow Compiler Invariants

This skill defines the non-negotiable rules of the Flow compiler codebase. Every rule here exists because violating it causes a category of bug that is extremely difficult to find and fix after the fact. Read this fully before touching any compiler file.

---

## The Pipeline Is Linear and One-Way

The compiler pipeline is:

```
source string
  → lexer.py        → list[Token]
  → parser.py       → Module (AST)
  → resolver.py     → ResolvedModule
  → typechecker.py  → TypedModule
  → lowering.py     → LModule
  → emitter.py      → str (C source)
  → driver.py       → shells out to clang
```

Information flows forward only. No pass reaches back to a previous pass. No pass calls into a later pass. If you find yourself needing information from a later pass in an earlier one, the design is wrong — stop and reconsider.

**Violation example (never do this):** The resolver calling the type checker to resolve an ambiguous name. Instead, resolve what can be resolved structurally, and leave ambiguous cases for the type checker to resolve with type information.

---

## AST Nodes Are Immutable After Parsing

The AST nodes defined in `compiler/ast_nodes.py` are set in stone the moment the parser produces them. Downstream passes **annotate** nodes; they never modify them.

- The resolver adds a `symbols: dict[ASTNode, Symbol]` side map to `ResolvedModule`.
- The type checker adds a `types: dict[ASTNode, Type]` side map to `TypedModule`.
- The lowering pass reads from both side maps and produces a completely new `LModule`.

If you need to attach data to an AST node in a downstream pass, add it to the appropriate side map. Never add fields to AST node dataclasses after Epic 3 is complete. Never mutate a field on an existing AST node anywhere.

**Why:** Mutating AST nodes makes it impossible to rerun passes (for incremental compilation, error recovery, or testing) and creates invisible coupling between passes.

## Name Mangling Is Centralized in mangler.py

Every C identifier generated by the emitter must come from `compiler/mangler.py`. The mangling rules are:

```
fl_<module_path>_<TypeName>_<method_name>

math.vector.Vec3       → fl_math_vector_Vec3
math.vector.Vec3.dot   → fl_math_vector_Vec3_dot
pipeline.orders.run    → fl_pipeline_orders_run
```

**Never** construct a C identifier with string formatting outside of `mangler.py`. Never hardcode a mangled name anywhere else in the codebase. If you need a new kind of generated name (e.g., for a closure frame struct, a stream state machine, or a temp variable), add a function to `mangler.py` that produces it.

**Why:** One inconsistency in mangling produces a linker error. Linker errors are silent during compilation and produce inscrutable runtime failures. Centralization means there is one place to audit and one place to fix.

The one exception: temp variable names within a single function (`_fl_tmp_1`, `_fl_tmp_2`, etc.) are generated by the emitter's `fresh_temp()` method and are scoped to that function. They don't go through the mangler because they don't need to be globally unique.

## All Errors Go Through errors.py

The only acceptable way to signal a compiler error is to raise a subclass of `FlowError` from `compiler/errors.py`. The subclasses are:

- `LexError` — invalid character, unterminated literal
- `ParseError` — unexpected token, syntax error
- `ResolveError` — undefined name, scope violation
- `TypeError` — type mismatch, exhaustiveness failure, purity violation
- `EmitError` — malformed LIR, should be rare

Every error must include: `message`, `file`, `line`, `col`. If you don't know the exact line and col (e.g., an internal consistency error in the lowering pass), use the line/col of the input expression that triggered it.

**Never:**
- `raise ValueError("something went wrong")`
- `raise Exception("...")`
- `print("error: ...")` then `sys.exit(1)`
- `assert False, "unreachable"` in production code paths (use `EmitError` instead)

**Why:** The driver catches `FlowError` and formats it with file/line/col in a consistent, user-friendly way. Everything else produces ugly tracebacks that look like bugs in the compiler itself.

## Each File Owns Exactly One Concern

| File | Owns | Does NOT own |
|------|------|--------------|
| `lexer.py` | source → tokens | Nothing semantic. No keywords have meaning here beyond being tokens. |
| `parser.py` | tokens → AST | No type information. No semantic validation beyond syntax. |
| `ast_nodes.py` | dataclass definitions | No logic, no methods that do computation, no imports from other compiler modules. |
| `resolver.py` | name → Symbol binding | No type inference. Names are resolved structurally. |
| `typechecker.py` | type inference and checking | No C-level concerns. No mangling. No LIR. |
| `lowering.py` | typed AST → LIR | No formatting. No C syntax. Produces LIR nodes only. |
| `emitter.py` | LIR → C string | No decisions. Pure formatting. If a decision is needed, it belongs in lowering. |
| `mangler.py` | Flow names → C identifiers | Nothing else. |
| `errors.py` | error type definitions | No logic. |
| `driver.py` | pipeline orchestration | No compiler logic. Calls the other modules in order. |

If you find yourself importing `lowering.py` from `resolver.py`, or `typechecker.py` from `emitter.py`, something is wrong.

## The Bootstrap Constraint

The Python compiler is temporary scaffolding. The self-hosted Flow compiler (Epic 11) must be able to replicate everything the Python compiler does. This means:

**Never use a Python-specific trick inside the compiler pipeline that has no Flow equivalent.** Specifically:

- Do not rely on Python's arbitrary-precision integers in a way the compiler logic depends on. Flow integers are bounded.
- Do not use Python introspection (`type()`, `isinstance()` deep chains, `__dict__`) as a shortcut in the pipeline. Use explicit dispatch on node types.
- Do not use Python dataclass inheritance in a way that the type system exploits at runtime. The self-hosted compiler won't have Python's MRO.
- Do use Python's `match` statement for dispatching on AST node types — this maps cleanly to Flow's `match` on sum types.

The runtime library (`flow_runtime.h`, `flow_runtime.c`) is permanent and is not subject to this constraint. It can use any C that compiles on the target platform.

## Tests Before Done

A ticket is not done until:

1. The feature works.
2. `make test` passes.
3. At least one test exercises the new behavior (positive case).
4. For any behavior that can fail, at least one test exercises the failure (negative case).

For emitter tickets, "test" means a golden file test: a `.flow` program compiles to C and the output exactly matches the expected file in `tests/expected/`. When you intentionally change emitter output, update the golden file deliberately, not automatically.

Do not mark a ticket complete and move to the next one without running the tests. Even if you are confident the code is correct, run the tests.

## When In Doubt, Check the Spec

`flow_spec.md` is the source of truth for all language semantics. When implementing any feature, read the relevant spec section before writing code. Do not implement what seems obvious — implement what the spec says.

The five most common places where "obvious" and "spec" diverge:

1. `:imut` vs `:mut` parameter passing rules (`:imut` accepts both, `:mut` requires `:mut` or `@copy`)
2. Auto-lifting for `option<T>` (only when target type is statically known and non-generic)
3. Struct spread `..source` bypasses constructors (no validation)
4. `ex.data` is mutable in retry blocks; `ex.original` is always immutable
5. Stream single-consumer: a stream binding consumed once is consumed forever

If the spec is silent on a behavior, do not invent one. Flag it as a spec gap and note it in a comment.
